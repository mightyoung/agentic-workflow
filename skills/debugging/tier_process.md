## 进入条件

- 收到明确的 bug 报告或问题描述
- 遇到执行失败、错误异常或不符合预期的情况
- 从其他阶段转来的需要调试的问题

## 轻量 vs 深度调试决策

| 场景 | 调试档位 |
|------|----------|
| 单文件、低依赖、错误信息明确 | 轻量 `0% / 25%` |
| 多文件联动、失败历史重复、根因不清 | 升级 `50%` |
| `syntax_error` / `type_error` / `quality_gate_failed` / 重复 `test_failure` | 触发激活升级 |

## 核心 Triage 步骤

1. **Step 0.5 Reflexion 检索**：先查历史反思（causal search + full search），命中则直接参考已验证修复路径
2. **Step 1 闻味道**：收集错误信息、复现问题、确定问题域、评估紧急程度
3. **Step 2 揪头发**：5Why 分析 + ACH 竞争假设法，找到根本原因
4. **Step 3 照镜子**：审查最近 git 变更，过 7项检查清单（日志/边界/并发/资源/配置/依赖/环境）
5. **Step 4 执行**：最小修复 → 编写/更新测试 → 运行验证 → 交叉回归
6. **Step 5 复盘**：记录根因、修复方案、预防措施
7. **Step 5.5 Reflexion Entry**：写入结构化 verbal 反思到长期记忆（极简 typo 可跳过）

## 升级触发条件

- 1次失败 → 继续下一步
- 2次失败 → 切换调试策略
- 3次失败 → 进入深度分析：搜索 + 读源码
- 4次失败 → 完整 7项检查
- 5次及以上 → 最小 PoC 验证

## 退出条件

以下三项必须全部满足：
1. 根本原因已用具体 file:line 证据确认（不是猜测）
2. 修复后相关测试全部通过（必须运行并看到输出）
3. 复盘总结已记录（根因 + 修复方案 + 预防措施）

## MAGMA 因果搜索集成

Step 0.5 中通过 `memory_longterm.py --op search-causal` 查询 MAGMA 因果图；有命中时 Fix 字段即为已验证方案，可跳过相同失败路径，避免重复踩坑。
