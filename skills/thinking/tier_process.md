# THINKING — 过程路由与决策逻辑

## Entry Criteria（进入 THINKING 的条件）

- 用户提出需要深度分析的问题
- 出现"谁最懂"、"专家"、"分析"等关键词
- 需要理解底层逻辑或顶层设计
- 涉及安全、性能等专项分析
- PUA 激励触发（失败时自动进入）

## Exit Criteria（有效 THINKING 输出的判定）

退出 THINKING 阶段需满足：
- 完成 2-3 个专家视角的具体分析
- 得出明确推理结论（调查结论 + 主要矛盾 + 阶段判断 + 局部攻坚点）
- 识别出关键风险点和优化点
- 提供可执行的推荐行动

## 三类工作流 — 如何选择

| 工作流 | thinking_mode | 适用场景 |
|--------|---------------|----------|
| 新项目启动 | `investigation_first` | 方向未定、技术选型、全新系统 |
| 复杂问题攻坚 | `contradiction_analysis` | Bug 难以定位、性能瓶颈、架构冲突 |
| 方案迭代优化 | `mass_line_iteration` | 已有方案需改进、重构、用户反馈处理 |

**选择规则**：优先看问题是否有已知基础——有则走 `mass_line_iteration`；有已知基础但存在深层矛盾则走 `contradiction_analysis`；无已知基础从零出发走 `investigation_first`。

## 各工作流核心步骤

### investigation_first（新项目启动）

1. **调查研究**：读代码、日志、git history，汇总事实与约束，明确哪些是事实、哪些是推测
2. **矛盾分析**：列出最突出的对立面，找出主要矛盾的切入点
3. **群众路线**：多源汇聚（测试、代码库、日志、历史记录），系统化后再给出结论
4. **持久战略**：判断当前阶段（防御/相持/反攻），识别最小局部攻坚点

### contradiction_analysis（复杂问题攻坚）

1. **多路径探索**（ToT，复杂度 M/L/XL）：生成 3 个候选方向，加权评分，选最优 Branch
2. **第一性原理分析**：识别基本事实 → 质疑假设 → 从零构建方案
3. **矛盾聚焦**：找出根本矛盾，集中力量突破主要矛盾的主要方面
4. **批评自我批评**：检查分析框架的盲点，考虑反面证据

### mass_line_iteration（方案迭代优化）

1. **群众路线**：收集所有来源的反馈（测试结果、用户报告、代码审查意见）
2. **矛盾分析**：找出当前方案的核心矛盾（为什么改了还是不对？）
3. **实践认知**：对比上一轮迭代的效果，判断变化趋势
4. **持久战略**：判断迭代方向是否收敛，更新阶段判断

## Thinking Mode 选择逻辑

```
IF 问题方向未定 OR 全新系统:
    → thinking_mode = investigation_first
    → thinking_methods = [调查研究, 矛盾分析, 群众路线, 持久战略]

ELSE IF 存在深层矛盾 OR 难以定位的失败:
    → thinking_mode = contradiction_analysis
    → thinking_methods = [调查研究, 矛盾分析, 集中力量, 实践认知, 批评自我批评]

ELSE IF 已有方案需改进:
    → thinking_mode = mass_line_iteration
    → thinking_methods = [群众路线, 矛盾分析, 实践认知, 批评自我批评]
```

## 完成状态

| 状态 | 含义 | 触发条件 |
|------|------|----------|
| **DONE** | 完成 | 完成所有专家视角分析，得出明确结论 |
| **DONE_WITH_CONCERNS** | 带担忧完成 | 存在未解决的技术风险，但不影响整体方向 |
| **BLOCKED** | 阻塞 | 缺少关键上下文，无法继续推理 |
| **NEEDS_CONTEXT** | 需要更多信息 | 需要用户提供额外信息才能继续 |
