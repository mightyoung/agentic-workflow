# THINKING — 完整方法论

## 求是式四步法（完整版）

### Step 0.5: 调查研究 — 把事实先收齐

借鉴 `investigation-first` 的思路，在下判断前先把第一手材料集中起来：

- 读代码、测试、日志、git history
- 汇总用户反馈和当前约束
- 明确哪些是事实，哪些只是推测

**输出**：
```markdown
## 调查结论
- 现状是：...
- 关键约束是：...
- 主要未知项是：...
- 已知可依托基础是：...
```

### Step 0.75: 矛盾分析 — 找主要矛盾

借鉴 `contradiction-analysis` 的思路，把调查到的事实按冲突关系重新组织：

- 列出当前最突出的对立面
- 找出哪个矛盾最能解释其他问题
- 判断主要矛盾的主要方面

**输出**：
```markdown
## 主要矛盾
- [A] vs [B]
- 矛盾性质：[对抗性/非对抗性]
- 最值得突破的切入点：...
```

### Step 1.5: 群众路线 — 多源汇聚再验证

借鉴 `mass-line` 的方法，把分散的信息源集中起来再返回验证：

- 来源不限于用户，还包括测试、代码库、日志、历史记录
- 不要闭门造车，也不要机械照搬单一来源
- 把多个来源的意见系统化后再给出结论

**输出**：
```markdown
## 多源反馈
- [信息源1]：...
- [信息源2]：...
- 各源之间的矛盾之处：...
- 共同指向的问题：...
```

### Step 1.75: 持久战略 — 判断阶段与局部攻坚点

借鉴 `protracted-strategy` 的思路，把 THINKING 的结论转换成阶段判断：

- 判断当前是战略防御、战略相持还是战略反攻
- 识别一个最小的局部攻坚点
- 明确什么时候该进入 PLANNING / EXECUTING

**输出**：
```markdown
## 阶段判断
- 当前阶段：...
- 局部攻坚点：...
- 转入下一阶段的条件：...
```

**阶段判断标准**：
- **战略防御**：问题本质未明，资源不足以正面突破，先稳住、摸清情况
- **战略相持**：已知主要矛盾，但尚未找到决定性突破口，持续施压+局部试探
- **战略反攻**：主要矛盾已明确，具备条件发起系统性解决，全面推进

## 多路径探索（ToT — 复杂度 M/L/XL）

> 对标 Tree of Thoughts (arXiv 2305.10601)：在深度分析之前，先做一次宽度为 3 的浅层 BFS，避免过早锁定单一路径导致局部最优。

**触发条件**：复杂度 M/L/XL，或存在多种可行方向，或架构/技术选型类问题。
**跳过条件**：复杂度 XS/S，或问题方向已高度明确（无争议）。

### Layer 1：3 个候选方向

每个 Branch 评估：核心思路、优点（2点）、主要风险（1点）、可行性评分（0-10）。
Branch C 建议来自第一性原理（忘掉现有方案从零构建）。

**量化评分标准**（必须输出数值）：

| 维度 | 权重 | 0-10 评分标准 |
|------|------|---------|
| 技术可行性 | 40% | 0=不可能, 5=需大量改造, 10=现有栈直接支撑 |
| 完整性 | 30% | 0=仅覆盖边缘, 5=覆盖核心, 10=核心+边界+异常全覆盖 |
| 风险 | 30% | 0=失败不可逆, 5=失败代价中等, 10=可快速回滚 |

```
Score(Branch) = 可行性×0.4 + 完整性×0.3 + 风险×0.3
```

### Layer 2：选择最优分支

- 最高分 Branch 进入深度分析
- 如两个 Branch 分差 ≤1.0，**两个都保留**做并行分析后再选

### 回溯检测（Backtrack Gate）

回溯触发条件（满足任一则回溯）：

| # | 条件 |
|---|------|
| 1 | 深度分析发现不可绕过的技术约束 |
| 2 | 安全专家判定存在不可缓解的安全风险 |
| 3 | 第一性原理分析推翻了 Branch 的核心假设 |

```
IF 致命缺陷被发现:
  1. 记录: "[BACKTRACK] Branch X 被放弃，原因: {一句话}"
  2. 切换到次优 Branch Y，从深度分析重新开始
  3. 如果 Branch Y 也失败 → 回到 Layer 1 生成新 Branch D
  注意: 最多回溯 2 次，超过则输出 BLOCKED 状态
```

## 第一性原理分析

**三步法**：

1. **识别基本事实 (Identify Axioms)**：从用户需求中提取不可辩驳的事实——核心问题、技术约束、业务边界。输出：基本事实清单。

2. **质疑假设 (Challenge Assumptions)**：对每个隐含假设追问"这个假设是真的吗？如果没有这个假设会怎样？在什么条件下不成立？"。常见被质疑假设："必须用 XX 技术栈"、"现有架构不能改变"、"用户知道他们想要什么"。

3. **从零构建 (Build from Scratch)**：忘掉现有方案——这个问题最简单可工作的解决方案是什么？行业最佳实践是否适用于这个场景？输出：从零方案。

**检查清单**：
- [ ] 是否识别了基本事实（Axioms）？
- [ ] 是否列出了隐含假设并逐一质疑？
- [ ] 是否尝试了从零构建？
- [ ] 最终方案是否优于从零方案？

## 专家角色选择（动态，非固定）

根据任务领域选择 2-3 个最相关的专家角色：

| 角色 | 适用场景 |
|------|----------|
| security_expert | 涉及认证、加密、输入验证 |
| performance_expert | 高并发、大数据、响应时间 |
| senior_engineer | 代码质量、可维护性、重构 |
| architecture_expert | 系统设计、模块划分、技术选型 |
| dba_expert | schema 设计、查询优化、数据迁移 |
| devops_expert | 部署、CI/CD、容器化 |
| ux_expert | UI/UX、可用性、交互设计 |
| domain_expert | 特定业务逻辑（金融、医疗、电商等） |

**选择原则**：只选与当前任务直接相关的 2-3 个角色，不要为凑数选不相关的角色。

## Memory 集成

THINKING 阶段应检索历史经验以避免重复踩坑：

**memory_hints 检索时机**：
- 进入 THINKING 之前，先检索相关历史分析结论
- 关键词：当前任务涉及的技术栈、问题类型、领域

**experience_check 要点**：
- 过去在相似问题上有哪些已验证的结论？
- 有哪些被推翻的假设记录在案？
- 哪些局部攻坚点曾经失败过？

**research_inputs**：
- 将调查研究阶段收集的第一手资料作为输入
- 包含：代码分析结论、测试结果摘要、日志关键发现

## Confidence Level 置信度判定

| 置信度 | 条件 |
|--------|------|
| 90-100% | 所有假设已验证，无未知项，有清晰的局部攻坚点 |
| 70-89% | 主要矛盾明确，少量未知项，攻坚点可执行 |
| 50-69% | 主要矛盾初步判断，存在若干未知项，需要试探性攻坚 |
| <50% | 事实不足，矛盾不清，需要更多调查才能继续 |

置信度 <50% 时，状态应输出 `NEEDS_CONTEXT` 或 `BLOCKED`。
