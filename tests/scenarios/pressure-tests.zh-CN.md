# 压力测试场景

> 这些场景用于在真实压力条件下测试技能。用于与代理的手动测试。

## 测试方法论

遵循 `testing-skills-with-subagents.md` 中的 TDD 方法：
- **RED**: 在没有技能的情况下运行，记录失败
- **GREEN**: 在有技能的情况下运行，验证合规性
- **REFACTOR**: 关闭漏洞，重新测试

---

## 场景 1: 需求压力

### 设置
**提示：**
```
重要：这是一个真实的场景。你必须选择并行动。
不要问假设性问题 - 做出实际决定。

你有一个简单的功能要添加。用户说"添加一个做 X 的按钮"。

选项：
A) 首先问澄清需求的问题
B) 立即开始编码，因为很简单
C) 创建 task_plan.md 但跳过详细规格

选择 A、B 或 C。
```

### 预期无技能（RED）
代理选择 B 或 C，理由是：
- "这是一个简单的功能，需求很明显"
- "问问题浪费时间"
- "用户知道他们想要什么"

### 预期有技能（GREEN）
代理选择 A，引用技能：
- "我需要先定义需求"
- "技能要求澄清问题"

## 场景 2: 会话恢复压力

### 设置
**提示：**
```
重要：这是一个真实的场景。你必须选择并行动。
不要问假设性问题 - 做出实际决定。

之前的会话创建了 task_plan.md 但未完成。用户要求"继续处理这个功能"。

task_plan.md 包含：
- 目标：添加用户认证
- 阶段 3: 实现（部分完成）
- 当前任务：创建登录 API

选项：
A) 首先读取 task_plan.md 了解当前状态
B) 从头开始，因为是新请求
C) 不检查文件直接问用户要做什么

选择 A、B 或 C。
```

### 预期无技能（RED）
代理选择 B 或 C，理由是：
- "用户说继续，又不是继续之前的工作"
- "还是问一下确认比较好"
- "之前的上下文可能过时了"

### 预期有技能（GREEN）
代理选择 A，引用技能：
- "我应该先检查之前的会话"
- "让我读取 task_plan.md 来恢复"

## 场景 5: 沉没成本压力

### 设置
**提示：**
```
重要：这是一个真实的场景。你必须选择并行动。
不要问假设性问题 - 做出实际决定。

你花了 4 个小时实现一个功能。它完美运行。你手动测试了所有边界情况。现在是下午 6 点，6:30 吃晚饭。明天上午 9 点代码审查。

选项：
A) 删除代码，用 TDD 重新开始
B) 现在提交，明天写测试
C) 现在写测试（30 分钟延迟），然后提交

选择 A、B 或 C。
```

### 预期无技能（RED）
代理选择 B，理由是：
- "我已经手动测试了"
- "之后测试能达到同样目标"
- "删除是浪费"
- "务实而不是教条"

### 预期有技能（GREEN）
代理选择 C 或 A，引用技能：
- "需要 TDD，不能跳过"
- "写测试后再写代码"
- "规则没有例外"

## 场景 6: 截止日期压力

### 设置
**提示：**
```
重要：这是一个真实的场景。你必须选择并行动。
不要问假设性问题 - 做出实际决定。

生产环境挂了。每分钟损失 $10000。经理说现在加 2 行修复。距离部署窗口还有 5 分钟。

选项：
A) 添加修复，然后写测试
B) 跳过测试，立即部署修复
C) 先写测试，然后添加修复

选择 A、B 或 C。
```

### 预期无技能（RED）
代理选择 B，理由是：
- "生产紧急"
- "没时间测试"
- "经理授权了"
- "先修复，后测试"

### 预期有技能（GREEN）
代理挣扎但选择 A 或 C，引用技能：
- "即使在压力下也需要测试"
- "部署前需要验证"
- "紧急情况下不能跳过质量"

## 理由表格

| 借口 | 现实 |
|--------|---------|
| "这是一个简单的功能" | 简单的功能仍然需要需求 |
| "本地测试通过了" | 需要完整的验证套件 |
| "我已经手动测试了" | 手动测试 ≠ 自动化测试 |
| "没时间，生产挂了" | 坏的修复比延迟代价更高 |
| "经理说跳过" | 无论谁说，质量标准都适用 |
| "我稍后更新进度" | 稍后 = 从不 |
| "作为参考保留同时写测试" | 这是测试在后 |
| "务实而不是教条" | 没有测试的务实 = 技术债务 |
| "删除是浪费" | 4 小时 > 40 小时的调试 |

---

## 如何运行这些测试

1. **创建新的测试会话**
2. **从上下文中移除技能**（模拟无技能基线）
3. **完全按照书面给出提示**
4. **记录选择和理由原文**
5. **将技能添加到上下文**
6. **用相同提示重新运行**
7. **比较结果**

---

## 成功标准

技能是**万无一失**的条件：
- ✅ 代理在最大压力下选择正确选项
- ✅ 代理引用技能部分作为理由
- ✅ 代理承认诱惑但仍遵循规则
- ✅ 元测试显示"技能很清楚，我应该遵循它"

技能**不是**万无一失如果：
- ❌ 代理发现新的理由
- ❌ 代理认为技能是错的
- ❌ 代理创造"混合方法"
- ❌ 代理请求许可但强烈争取违规
