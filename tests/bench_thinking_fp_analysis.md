**THINKING 阶段流程**：

## 案例对比

### 场景: 技术栈选型

**用户请求**: "我们应该用 React 还是 Vue？"

#### 改进前分析

```
专家视角: 资深前端工程师
- React: 生态大、社区活跃、Facebook 背书
- Vue: 上手简单、中文文档好、国人使用多

结论: 根据团队技术背景选择
```

#### 改进后分析

```
Step 0: 第一性原理分析
├── 基本事实 (Axioms):
│   ├── 团队: 5人，前端为主
│   ├── 场景: 企业内部系统
│   ├── 约束: 需要长期维护
│   └── 核心: UI 组件复用
│
├── 质疑假设:
│   ├── ❌ "React 生态最大" → 内部系统不需要生态
│   ├── ❌ "Vue 上手简单" → 5年经验团队这不是问题
│   ├── ❌ "必须二选一" → 可以渐进迁移
│   └── ❌ "框架决定一切" → 架构设计比框架更重要
│
└── 从零构建:
    核心问题: 如何高效构建可复用 UI 组件？
    → Web Components 是不是更本质的解决方案？
    → 微前端架构是否更符合长期维护需求？

Step 1: 专家视角分析
├── 架构师: 微前端 + 模块化
├── 安全专家: 依赖漏洞管理
└── 性能专家: 首屏加载、包体积

结论: 建议渐进式，先用原生 Web Components 构建基础组件库
```

## 结论

### 改进效果

1. **分析深度提升**: 新增第一性原理分析步骤，从"专家视角"升级到"基本原则"
2. **假设审查机制**: 显式质疑隐含假设，避免思维盲区
3. **创新空间打开**: "从零构建"鼓励跳出现有方案
4. **与其他项目差异化**: 明确使用第一性原理方法论

### 适用性建议

| 任务复杂度 | 是否启用 | 原因 |
|------------|----------|------|
| 简单任务 | 跳过 | 过度设计 |
| 中等任务 | 可选 | 根据是否涉及重大决策 |
| 复杂/创新任务 | **强烈建议** | 第一性原理价值最大 |

### 下一步优化方向

1. 增加第一性原理与专家视角的权重配置
2. 添加"假设质疑清单"模板
3. 建立从零构建方案数据库
