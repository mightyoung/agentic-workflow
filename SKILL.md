# Agentic Workflow - 统一智能体工作流

> 融合 best-minds, brainstorming, writing-plans, planning-with-files, TDD, systematic-debugging, verification, pua, tavily 精髓

## 核心原则

### 1. 专家模拟思维 (Best-Minds)
不要问"你怎么看"，而是问"这个问题谁最懂？TA会怎么说？"。

### 2. 文件持久化 (Planning-with-Files + RESEARCH)
- task_plan.md - 任务计划
- findings.md - 研究发现
- progress.md - 进度追踪

### 3. TDD 驱动
测试先行 → 失败 → 实现 → 通过

### 4. RESEARCH 前置搜索 (Tavily)
- 思考前先搜索最佳实践
- 规划前先调研成熟案例
- 将搜索结果存入 findings.md

### 5. PUA 激励引擎 (失败时触发)

### 6. 注意力管理
- **铁律一：穷尽一切** - 没有穷尽所有方案之前，禁止说"无法解决"
- **铁律二：先做后问** - 遇到问题先自行搜索、读源码、验证，再提问
- **铁律三：主动出击** - 端到端交付，不只是"刚好够用"

### 5. 注意力管理
每3个动作循环重读 task_plan.md

---

## 状态机

```
IDLE → RESEARCH → THINKING → PLANNING → EXECUTING → REVIEWING → COMPLETE
              ↓           ↓           ↓           ↓
         DEBUGGING ←────────────────────────────────────→
```

### RESEARCH 阶段（自动触发）
- 触发条件：复杂任务（3+步骤）、新技术领域、需要外部参考
- 搜索范围：网络最佳实践、GitHub 成熟项目、社区讨论、官方文档
- 输出：结构化搜索结果存入 findings.md

## 路由逻辑

根据用户输入自动路由到对应模块：

```python
# 不触发 - 简单闲聊问题
if 包含("天气", "笑话", "你好", "hi", "hello") and not 包含("开发", "代码", "实现"):
    → 不触发，直接回答

# RESEARCH 触发 - 复杂任务自动搜索
if 包含("怎么做", "如何实现", "最佳实践", "有什么", "有哪些", "参考", "案例"):
    → 研究模块 (RESEARCH) → THINKING

# PUA 触发 - 失败时自动激活
if 包含("尽力", "别放弃", "继续尝试", "为什么还不行", "你再试试"):
    → 激活 PUA 模式（增强执行/调试）

elif 包含("谁最懂", "专家", "顶级", "best minds"):
    → 专家模拟模块 (THINKING)

elif 包含("计划", "规划", "拆分任务"):
    → 规划模块 + task_plan.md (PLANNING)

elif 包含("bug", "错误", "调试", "修复"):
    → 调试模块 + PUA 5步方法论 (DEBUGGING)

elif 包含("审查", "review", "检查spec"):
    → 审查模块 (REVIEWING)

else:
    → 执行模块 + PUA铁律 (EXECUTING)
```

## 模块详解

### 1. 研究模块 (RESEARCH)

**目的**：在思考和规划前，先搜索网络上、社区中、GitHub 中的最佳实践和成熟案例。

**触发条件**：
- 复杂任务（3+步骤）
- 新技术领域
- 需要外部参考
- 触发词："怎么做"、"如何实现"、"最佳实践"、"有什么"、"参考"、"案例"

**搜索策略**：
1. **网络搜索**（Tavily MCP）：
   - 官方文档和教程
   - 行业最佳实践
   - 技术博客和文章

2. **GitHub 搜索**：
   - 成熟开源项目
   - 最佳实践模板
   - 热门解决方案

3. **社区搜索**：
   - Reddit/Hacker News 讨论
   - Stack Overflow 高票答案
   - 开发者社区经验

**输出到 findings.md**：
```markdown
# 研究发现

## 网络最佳实践
- [来源1]: 关键要点
- [来源2]: 关键要点

## GitHub 成熟项目
- 项目A: 特点描述
- 项目B: 特点描述

## 社区经验总结
- 经验1
- 经验2
```

**自动进入下一阶段**：带着搜索结果进入 THINKING

### 2. 专家模拟模块 (THINKING)

**一步一步思考分析：**

#### 链式推理步骤（必做）
1. **定义问题**：用一句话说清楚要解决什么
2. **拆解要素**：这个问题由哪些子问题组成？
3. **验证前提**：每个前提是否为真？
4. **推导链条**：A→B→C，每步必须有充分理由
5. **寻找反例**：有没有推翻这个结论的例子？
6. **得出结论**：用一句话总结，必须包含条件

#### 专家模拟步骤（可选）
- 当需要外部视角时：
  1. 谁最可能解决过这个问题？
  2. TA 会怎么思考？
  3. 引用 TA 的公开言论

#### 输出格式
```markdown
## 思考分析

### 问题定义
[一句话]

### 要素拆解
- 要素1：xxx
- 要素2：xxx

### 推理链条
1. 因为 [前提A]
2. 所以 [中间结论B]
3. 因此 [最终结论C]

### 反例检验
- 反例1：[描述] → [反驳/成立]
- 反例2：[描述] → [反驳/成立]

### 结论
[带条件的结论]
```

### 3. 规划模块 (PLANNING)

1. **创建文件**
   - task_plan.md
   - findings.md
   - progress.md

2. **任务拆分**
   - 最小粒度: 2-5分钟
   - 每个任务独立可测试
   - 明确文件边界

3. **模板**
```markdown
# [任务名] 实施计划

> 专家视角: [谁最懂这个？TA会怎么说？]

## 目标
> 一句话描述

## 阶段

### Phase 1: [名称]
- [ ] 任务1
- [ ] 任务2

## 进度
- [Phase 1: 0%]

## 注意力提醒
> 每3步重读此文件
```

### 4. 执行模块 (EXECUTING)

**TDD 循环 + PUA铁律：**
1. 写失败测试
2. 运行确保失败
3. 最小代码实现
4. 运行确保通过
5. 提交

#### PUA铁律（强制执行）

**铁律一：穷尽一切**
- 测试失败后，必须尝试至少3种本质不同的解决方案
- 禁止在尝试2种方案后说"无法解决"

**铁律二：先做后问**
- 遇到问题先自行搜索、读源码、验证
- 向用户提问前必须先附上已排查的证据

**铁律三：主动出击**
- 修复后主动检查同类问题
- 完成后验证结果正确性
- 主动补充未明确说但合理的需求

#### 主动出击清单（每次实现后强制自检）

完成任何功能实现后，必须逐项检查：

- [ ] 实现是否经过验证？（运行测试、实际执行）
- [ ] 同文件/同模块是否有类似问题？
- [ ] 上下游依赖是否受影响？
- [ ] 是否有边界情况没覆盖？
- [ ] 是否有更好的方案被忽略？
- [ ] 用户没有明确说的部分是否主动补充？

#### 压力升级机制

当任务失败时，根据失败次数执行：

| 失败次数 | 等级 | 动作 |
|---------|------|-----|
| 2次 | L1 | 停止当前思路，切换本质不同的方案 |
| 3次 | L2 | 搜索完整错误+读源码+列出3个假设 |
| 4次 | L3 | 执行7项检查清单，列出3个全新假设 |
| 5次+ | L4 | 最小PoC+隔离环境+完全不同的技术栈 |

**子任务分发：**
- 独立任务 → 新子agent
- 传递精确上下文
- 两阶段审查

### 5. 审查模块 (REVIEWING)

**一步一步思考分析：**

1. 先看整体架构：模块划分是否清晰？
2. 再看核心逻辑：关键路径有没有漏洞？
3. 然后看边界条件：异常处理是否完善？
4. 接着看性能瓶颈：有没有 O(n²) 或内存泄漏？
5. 最后看安全风险：有没有注入、越权、泄露？
6. 每个问题必须给出具体代码位置和修复建议

#### 问题分级标准
| 级别 | 定义 | 处理方式 |
|-----|------|---------|
| 🔴 致命 | 导致系统崩溃、数据丢失、安全漏洞 | 必须修复，禁止合并 |
| 🟡 严重 | 功能错误、性能严重下降、API破坏 | 必须修复，可暂缓 |
| 🟢 建议 | 代码风格、可读性、优化建议 | 可选改进 |

#### 审查输出模板
```markdown
## 审查结论

### 致命问题 (0)
无

### 严重问题 (2)
1. [问题描述]
   - 位置: `src/auth.py:42`
   - 影响: 导致会话劫持
   - 修复建议: [具体代码]

### 建议 (3)
1. [建议描述]
   - 位置: `src/utils.py:15`
   - 优化方向: [建议]
```

#### 自我检验清单
- [ ] 我是否因为用户是"上级"而降低了问题严重程度？
- [ ] 我是否因为代码"能跑"而忽略了设计问题？
- [ ] 我是否因为"时间紧"而跳过了安全检查？
- [ ] 我是否在描述中使用了"可能"、"或许"等模糊词？

**规范检查 (OpenSpec 可选)：**

方案A - 使用 OpenSpec:
1. 检查 openspec/changes/<name>/proposal.md
2. 验证实现是否符合 spec
3. 参考 design.md

方案B - 使用 Markdown:
1. 读取 task_plan.md
2. 验证每个任务是否完成
3. 检查是否符合目标

**质量检查：**
- 代码风格
- 潜在bug
- 性能问题
- 测试覆盖

**不留情面原则：**
- 审查时只说真话，不迎合谄媚
- 问题是问题，优点是优点，分开陈述
- 用户不爱听的话更要直说
- 温和的语气是专业，温和的内容是失职
- "这个设计有问题" > "可能可以改进"

### 6. 调试模块 (DEBUGGING)

**根因分析 + PUA 5步方法论：**

#### Step 1: 闻味道 — 诊断卡壳模式
停下来。列出所有尝试过的方案，找共同模式。
如果一直在做同一思路的微调 = 原地打转。

#### Step 2: 揪头发 — 拉高视角（强制执行）
1. **逐字读失败信号** - 90%的答案直接被忽略
2. **主动搜索** - 让工具告诉你答案
3. **读原始材料** - 不是摘要，是源码/原文
4. **验证前置假设** - 所有假设用工具确认
5. **反转假设** - 从对立方向重查

#### Step 3: 照镜子 — 自检
- 是否在重复同一思路的变体？
- 是否只看了表面症状没找根因？
- 是否该搜索却没搜？该读文件却没读？

#### Step 4: 执行新方案
- 和之前的方案本质不同
- 有明确的验证标准
- 失败时能产生新信息

#### Step 5: 复盘
- 哪个方案解决了？为什么之前没想到？
- 主动延伸：检查同类问题、修复完整性、预防措施

#### 7项检查清单（L3+ 强制完成）

当调试失败3次以上时，必须逐项完成：

- [ ] **读失败信号**：逐字读完了吗？
- [ ] **主动搜索**：用工具搜索过核心问题了吗？
- [ ] **读原始材料**：读过失败位置的原始上下文了吗？
- [ ] **验证前置假设**：所有假设都用工具确认了吗？
- [ ] **反转假设**：试过与当前方向完全相反的假设吗？
- [ ] **最小隔离**：能在最小范围内隔离/复现这个问题吗？
- [ ] **换方向**：换过工具，方法、角度，技术栈、框架吗？

**错误保留：**
- 保留错误上下文
- 不立即删除失败尝试

## 完整工作流示例

```
用户: 帮我开发一个用户认证系统

1. THINKING: 这个问题谁最懂？(安全专家)
   → 添加专家视角到 task_plan.md

2. PLANNING: 创建计划文件
   → task_plan.md, findings.md, progress.md
   → 拆分任务：登录、注册、JWT

3. EXECUTING: TDD循环
   → 每个任务：测试→失败→实现→通过
   → 复杂任务分发子agent

4. REVIEWING: 规范检查
   → 验证是否符合 spec
   → 代码质量审查

5. COMPLETE: 更新文件状态
   → memory_store 存储经验
```

## 注意事项

1. **单一入口**：这个技能是唯一入口，避免激活多个skill
2. **KV-Cache**：保持prompt稳定，避免频繁刷新
3. **按需加载**：只在需要时加载特定模块
4. **错误保留**：调试时保留错误上下文
5. **PUA自动触发**：失败2次+自动进入增强模式
6. **穷尽原则**：没说"无法解决"之前，必须尝试3种方案
