# State Architecture

This document describes the state file architecture and consolidation roadmap.

## Current State Files

During workflow execution, the following state files are created:

| File | Type | Authoritative? | Purpose |
|------|------|----------------|---------|
| `.workflow_state.json` | JSON | **YES** | Single source of truth for workflow state |
| `.artifacts.json` | JSON | **YES** | Artifact registry (generated by unified_state) |
| `.task_tracker.json` | JSON | **YES** | Task tracking (managed by task_tracker.py) |
| `.contract.json` | JSON | Partial | Phase contract (managed by workflow_engine) |
| `progress.md` | Markdown | NO (derived) | Human-readable progress view |
| `SESSION-STATE.md` | Markdown | NO (derived) | Human-readable session state view |
| `task_plan.md` | Markdown | NO | Task breakdown plan |
| `trajectories/` | Directory | **YES** | Execution trajectory logs |

## Authoritative vs Derived Files

**Authoritative files** are the source of truth - code reads and writes these directly.

**Derived files** are generated from authoritative sources and should not be manually edited:
- `progress.md` - generated from `.workflow_state.json` phase info
- `SESSION-STATE.md` - generated from `.workflow_state.json` session info

## Consolidation Roadmap (P2-1)

### Phase 1: Document Architecture (Done)
- This document explains current state
- Clarifies authoritative vs derived files

### Phase 2: Stop Proactive Writes to Derived Files
**Target:** Make `progress.md` and `SESSION-STATE.md` generated on-demand

Currently, `workflow_engine.py` proactively writes to these files on each phase transition. Instead:
1. These files should be generated when explicitly requested
2. Code should read from `.workflow_state.json` only
3. Write functions should be deprecated

**Implementation approach:**
- Add `--op generate-views` to `workflow_engine.py` that generates markdown files from JSON
- Stop automatic writes in `initialize_workflow` and `advance_workflow`
- Existing files remain for backward compatibility but are no longer authoritative

### Phase 3: Merge Contract into Workflow State
**Target:** Merge `.contract.json` into `.workflow_state.json`

The contract data (goals, verification_methods, owned_files, failure_threshold) should be stored in the authoritative workflow state file rather than a separate JSON.

**Implementation approach:**
- Add `contract` field to `WorkflowState` dataclass
- `workflow_engine.py` writes contract to state instead of separate file
- Backward compatibility: if `.contract.json` exists but not in state, load and merge

## Current Read/Write Patterns

### Writes in workflow_engine.py
```python
# initialize_workflow
- .workflow_state.json (authoritative)
- .task_tracker.json (authoritative)
- progress.md (derived - should stop)
- SESSION-STATE.md (derived - should stop)
- task_plan.md (derived - plan template)

# advance_workflow
- .workflow_state.json (authoritative)
- .task_tracker.json (authoritative)
- progress.md (derived - should stop)

# complete_workflow
- .workflow_state.json (authoritative)
- .task_tracker.json (authoritative)
```

### Reads in workflow_engine.py
```python
# Most code reads from:
- .workflow_state.json (authoritative)
- .task_tracker.json (authoritative)

# Some code reads from:
- progress.md (derived - should migrate to .workflow_state.json)
- SESSION-STATE.md (derived - should migrate to .workflow_state.json)
```

## Recommendation

For immediate improvement without breaking changes:
1. Keep current file structure
2. Document that `progress.md` and `SESSION-STATE.md` are derived views
3. Future code should read from JSON files only
4. Consider adding `generate-views` CLI operation for on-demand markdown generation
