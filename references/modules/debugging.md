# 调试模块详解 (DEBUGGING)

> 根因分析 + PUA 5步方法论 + 7项检查清单

## PUA 5步方法论

### Step 1: 闻味道 — 诊断卡壳模式

停下来。列出所有尝试过的方案，找共同模式。

如果一直在做同一思路的微调 = 原地打转。

**诊断问题**：
- 是不是在重复同一方案的变体？
- 是不是只改了参数没改思路？
- 是不是每次失败都采用类似的方法？

### Step 2: 揪头发 — 拉高视角（强制执行）

1. **逐字读失败信号** - 90%的答案直接被忽略
   - 错误信息每个词都要理解
   - 警告信息不能忽略

2. **主动搜索** - 让工具告诉你答案
   - 搜索完整错误信息
   - 搜索相关技术栈问题

3. **读原始材料** - 不是摘要，是源码/原文
   - 看官方文档原始内容
   - 看框架/库的源码

4. **验证前置假设** - 所有假设用工具确认
   - "假设这个API是这样工作的"
   - 实际验证假设是否正确

5. **反转假设** - 从对立方向重查
   - "如果反过来呢？"
   - "如果不是这个原因呢？"

### Step 3: 照镜子 — 自检

- 是否在重复同一思路的变体？
- 是否只看了表面症状没找根因？
- 是否该搜索却没搜？该读文件却没读？

### Step 4: 执行新方案

- 和之前的方案本质不同
- 有明确的验证标准
- 失败时能产生新信息

### Step 5: 复盘

- 哪个方案解决了？为什么之前没想到？
- 主动延伸：检查同类问题、修复完整性、预防措施

## 7项检查清单（L3+ 强制完成）

当调试失败3次以上时，必须逐项完成：

- [ ] **读失败信号**：逐字读完了吗？
- [ ] **主动搜索**：用工具搜索过核心问题了吗？
- [ ] **读原始材料**：读过失败位置的原始上下文了吗？
- [ ] **验证前置假设**：所有假设都用工具确认了吗？
- [ ] **反转假设**：试过与当前方向完全相反的假设吗？
- [ ] **最小隔离**：能在最小范围内隔离/复现这个问题吗？
- [ ] **换方向**：换过工具，方法、角度，技术栈、框架吗？

## 错误保留原则

- 保留错误上下文
- 不立即删除失败尝试
- 记录每次尝试的错误信息

## 调试心态

- 失败是信息，不是失败
- 每次失败都排除一个错误假设
- 坚持到找到根因
