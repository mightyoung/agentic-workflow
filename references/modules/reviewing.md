# 审查模块详解 (REVIEWING)

> 代码质量审查 + 问题分级 + 规范检查

## 审查流程

1. 先看整体架构：模块划分是否清晰？
2. 再看核心逻辑：关键路径有没有漏洞？
3. 然后看边界条件：异常处理是否完善？
4. 接着看性能瓶颈：有没有 O(n²) 或内存泄漏？
5. 最后看安全风险：有没有注入、越权、泄露？
6. 每个问题必须给出具体代码位置和修复建议

## 问题分级标准

| 级别 | 定义 | 处理方式 |
|-----|------|---------|
| 🔴 致命 | 导致系统崩溃、数据丢失、安全漏洞 | 必须修复，禁止合并 |
| 🟡 严重 | 功能错误、性能严重下降、API破坏 | 必须修复，可暂缓 |
| 🟢 建议 | 代码风格、可读性、优化建议 | 可选改进 |

## 审查输出模板

```markdown
## 审查结论

### 🔴 致命 (N)
1. [问题] - 位置: `file:line` - 影响: [一句话] - 修复: [简短建议]

### 🟡 严重 (N)
1. [问题] - 位置: `file:line` - 影响: [一句话] - 修复: [简短建议]

### 🟢 建议 (N)
1. [建议] - 位置: `file:line`
```

> 注：保留"影响"字段确保专业性

## 自我检验清单

- [ ] 我是否因为用户是"上级"而降低了问题严重程度？
- [ ] 我是否因为代码"能跑"而忽略了设计问题？
- [ ] 我是否因为"时间紧"而跳过了安全检查？
- [ ] 我是否在描述中使用了"可能"、"或许"等模糊词？

## 规范检查

### 方案A - 使用 OpenSpec
1. 检查 openspec/changes/<name>/proposal.md
2. 验证实现是否符合 spec
3. 参考 design.md

### 方案B - 使用 Markdown
1. 读取 task_plan.md
2. 验证每个任务是否完成
3. 检查是否符合目标

## 质量检查项

- 代码风格（格式、命名、注释）
- 潜在bug（空指针、越界、竞态）
- 性能问题（复杂度、内存、IO）
- 测试覆盖（边界条件、异常路径）

## 不留情面原则

- 审查时只说真话，不迎合谄媚
- 问题是问题，优点是优点，分开陈述
- 用户不爱听的话更要直说
- 温和的语气是专业，温和的内容是失职
- "这个设计有问题" > "可能可以改进"
