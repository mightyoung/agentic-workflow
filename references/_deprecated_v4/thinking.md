# 思考模块详解 (THINKING)

> 专家模拟思维 + 链式推理 + 反例检验 + 分段设计确认

## 核心原则

不要问"你怎么看"，而是问"这个问题谁最懂？TA会怎么说？"。

## 分段设计确认机制 (v4.12新增)

### 设计呈现规则

**提问策略**：
- 一次只问一个问题
- 多选优于开放式
- 理解目的、约束、成功标准

**方案探索**：
- 每次提出 2-3 个方案
- 包含权衡分析和推荐方案
- 推荐方案放在最前面

**分段呈现**：
按复杂度分段，每段后确认：

```markdown
**第一部分：数据模型**
[描述...]

确认：这个数据模型看起来合理吗？
→ 等待用户回应

**第二部分：API 设计**
[描述...]

确认：API 设计符合你的预期吗？
→ 等待用户回应

**第三部分：错误处理**
[描述...]

确认：错误处理方案可以吗？
```

### 设计规模指导

| 项目类型 | 设计篇幅 | 确认节奏 |
|----------|----------|----------|
| 简单（单文件） | 几句话 | 1次确认 |
| 中等（2-3文件） | 每部分100-200字 | 每部分确认 |
| 复杂（多系统） | 每部分200-300字 | 每部分确认 |

### 设计文档要求

设计批准后，写入文档：
```bash
docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md
```

文档结构：
```markdown
# [主题] 设计文档

> 日期: YYYY-MM-DD

## 背景
[用户目标和约束]

## 设计决策

### 决策1：...
### 决策2：...

## 架构
[数据模型/组件关系]

## 接口
[API/函数签名]

## 错误处理
[边界情况和处理]

## 确认
- [x] 用户已批准设计
```

### 设计重审循环

如果用户要求修改：
1. 理解修改点
2. 更新设计
3. 重新获取批准
4. 更新设计文档

## 链式推理步骤（必做）

1. **定义问题**：用一句话说清楚要解决什么
2. **拆解要素**：这个问题由哪些子问题组成？
3. **验证前提**：每个前提是否为真？
4. **推导链条**：A→B→C，每步必须有充分理由
5. **寻找反例**：有没有推翻这个结论的例子？
6. **得出结论**：用一句话总结，必须包含条件

## 专家模拟步骤（可选）

当需要外部视角时：
1. 谁最可能解决过这个问题？
2. TA 会怎么思考？
3. 引用 TA 的公开言论

### 专家识别模式

| 问题领域 | 可能的专家 |
|----------|-----------|
| 系统架构 | 系统架构师、技术负责人 |
| 算法问题 | 算法工程师、学术研究者 |
| 安全问题 | 安全专家、渗透测试工程师 |
| 性能优化 | 性能工程师、SRE |
| 产品设计 | 产品经理、UX设计师 |

## 输出格式

```markdown
## 思考分析

### 问题定义
[一句话]

### 要素拆解
- 要素1：xxx
- 要素2：xxx

### 推理链条
1. 因为 [前提A]
2. 所以 [中间结论B]
3. 因此 [最终结论C]

### 反例检验
- 反例1：[描述] → [反驳/成立]
- 反例2：[描述] → [反驳/成立]

### 结论
[带条件的结论]
```

## 推理检查清单

- [ ] 问题定义是否清晰无歧义？
- [ ] 要素拆解是否完整？
- [ ] 每个前提是否经过验证？
- [ ] 推导链条每步是否有充分理由？
- [ ] 是否尝试寻找反例？
- [ ] 结论是否包含条件？
